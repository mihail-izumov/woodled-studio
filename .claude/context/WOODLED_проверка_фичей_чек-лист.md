# WOODLED — Чек-лист проверки фичей (для AI-ассистента)

> Версия: 1.0 | Дата: 5 июня 2026
> Назначение: методика верификации после имплементации фичи без запуска dev-сервера. В песочнице нет `node_modules`, типчек и dev-run недоступны — но методичная mental-симуляция + grep + AST-парсинг ловят 80% багов до того как они доедут до пользователя.
> Когда применять: после каждой нетривиальной фичи, особенно после правок, затрагивающих центральные структуры (`Fixture`, `MD`, `Room`, share, brightness). Базируется на реальной проверке фичи кастомных светильников.

---

## Когда нужна проверка

- Фича изменила тип `Fixture`, `Room` или другую центральную модель.
- Добавилось новое поле, которое должно ехать в URL-хеш (`share.ts`).
- Добавился новый рендер-путь в живой движок (forest/story/RoomCard/ZoneCard).
- Затронут store (`addFixture` / `updateFixture` / `updateRoom` / `loadFromHash` / `restorePersistedState`).
- Появилась новая «опциональная» сущность, которая может быть `undefined` (легко забыть fallback).

Если правки строго локальные (один компонент, один экран без перетекания данных в lefty) — можно ограничиться парсингом и точечным грепом.

## Шаг 1 — Парсинг (формальная проверка синтаксиса)

В песочнице доступен `node` + `typescript`. Не нужно `vue-tsc` — достаточно базового AST-парсинга `.ts` и `<script lang="ts">` блоков `.vue`. Ловит:
- незакрытые скобки/теги
- обрезанные импорты после rebase
- опечатки в синтаксисе `as Type` в шаблоне

```bash
cd /sessions/<session>/mnt/outputs && npm i --no-save typescript 2>&1 | tail -1
node -e "
const ts = require('typescript');
const fs = require('fs');
const files = [
  ['ts',  '/sessions/<session>/mnt/woodled-studio/.vitepress/customizer/...file.ts'],
  ['vue', '/sessions/<session>/mnt/woodled-studio/.vitepress/customizer/components/...file.vue'],
];
let fails = 0;
for (const [kind, f] of files) {
  let src = fs.readFileSync(f, 'utf8');
  if (kind === 'vue') {
    const m = src.match(/<script[^>]*lang=['\"]ts['\"][^>]*>([\s\S]*?)<\/script>/);
    if (!m) continue;
    src = m[1];
  }
  const sf = ts.createSourceFile(f, src, ts.ScriptTarget.Latest, true, ts.ScriptKind.TS);
  const diags = (sf.parseDiagnostics || []);
  if (diags.length) {
    fails++;
    console.log('FAIL', f.split('/').slice(-2).join('/'));
    for (const d of diags.slice(0, 3)) console.log('  ', ts.flattenDiagnosticMessageText(d.messageText, '\n'));
  }
}
console.log(fails === 0 ? 'all parse ok' : fails + ' failures');
"
```

Парсинг чист = синтаксис ОК. Семантику он не ловит — для неё mental-симуляция.

## Шаг 2 — Mental-симуляция по сценариям

Для каждого «потока данных» прогоняем сценарий «что если». Минимальный набор для фичи, которая добавляет fixture с новым типом данных:

### A. Добавление нового объекта

1. Пользователь нажимает кнопку добавления → какой emit? куда?
2. Кто создаёт начальный объект (placeholder)?
3. Где регистрируется в shared-store (MD, FX_FACTORS, что-то ещё)?
4. **Когда** регистрируется? До или после первого рендера? (Если после — может упасть на `MD[id] === undefined`.)
5. Открывается ли редактор сразу? С какими props? `isProvisional`?

### B. Заполнение и сохранение

6. Какие поля формы инициализируются из объекта? Какие из дефолтов?
7. `buildSpec()` (или эквивалент) возвращает то же что было — если пользователь ничего не менял?
8. `isDirty` корректен сразу после открытия? (Должен быть `false` для existing без правок, `true` для provisional.)
9. При сохранении: store.update → перерегистрация (если нужно) → persist?

### C. Просмотр результата в UI

10. Где этот объект отображается? (карточка, тэлли, бейдж, текст?)
11. Каждое поле объекта используется напрямую или через хелпер?
12. Хелперы (`fxType`, `fxLine`, `fxChip`, `orbStyle`) корректно возвращают значения для нового типа?
13. Пустые поля — как отображаются? Пустая строка рендерит пустой span — это нормально или ломает layout?

### D. Редактирование существующего

14. Открытие через клик → `isNew=false`, `isProvisional=false`.
15. Поля формы восстанавливаются из объекта точно?
16. UI-state (toggle'ы, выбранные мощности, длины) — где сохраняются? восстанавливаются?
17. После «Сохранить» → правки применяются. После «Назад» без сохранения → правки откатываются.

### E. Удаление

18. Удаление через кнопку → confirm модалка → `emit('delete')` → store.removeFixture.
19. Удаление через «Назад» в provisional → store.removeFixture.

### F. Шаринг (если объект сериализуется)

20. `share.ts` имеет поле в `Packed*`? Pack кладёт, unpack восстанавливает?
21. ID объекта стабилен между отправителем и получателем?
22. После unpack — регистрация в store происходит ДО первого рендера?
23. Старые ссылки (legacy без нового поля) ещё работают?

### G. Persist / restore из localStorage

24. `restorePersistedState` обрабатывает новые поля?
25. `persistState` сохраняет их корректно?
26. На старте app — все рантайм-регистрации происходят ДО ensureStarterRooms / ensureWelcome?

### H. Влияние на «карточки настроения» / Story

27. Какие места в forest/story/RoomCard читают поле из fixture?
28. Если новый тип — лес читает только WOODLED (фильтр `!f.custom`) или включает?
29. Группировка (`groupByModel`) — даёт ли разумный результат?
30. Тексты карточек (через `fxNameNom`/`fxNameGen`) корректны?

### I. Лимиты, валидации

31. Лимит зоны проверяется в `addFixture`. А в `updateFixture` (если смена зоны)?
32. Если поле может быть `0` или `undefined` — формула не делит на ноль?
33. Если поле может быть строкой произвольной длины — обрезается до maxLength?

## Шаг 3 — Grep-приёмы для скрытых багов

### A. Поиск всех мест чтения поля

```
Grep "fx\.opts" → если ввели новое опциональное поле в Fixture, оно может конкурировать с opts
Grep "MD\[.*\.m\]|MD\[fx\.m\]" → 48 lookups в проекте, все должны работать
Grep "\.wood" → если поменяли wood-семантику
Grep "f\.q ??" → правильно ли учтён quantity
```

### B. Поиск edge cases в сериализации

```
Grep "packFixture|unpackFixture|PackedFixture" → все ли поля прокинуты
Grep "persistState|restorePersisted" → синхронность с share
```

### C. Поиск регистраций / реактивности

```
Grep "rooms\.splice|rooms\[.*\] =" → потенциальные точки рендера
Grep "registerCustom|registerAll" → все вызовы покрыты?
```

## Шаг 4 — Типичные баги (по реальному опыту)

### Баг класса 1: «опциональное поле = undefined»

Новое поле `Fixture.foo?: Bar`. Старый код читает `fx.foo.x` → crash.

**Где искать:**
- В `data/catalog.ts` хелперы типа `fxType(m)` читают `MD[m].x` — может ли `m` существовать с `undefined.x`?
- В `engine/*.ts` все `f.opts?.btemp` — есть ли у нового типа объекта `opts`? Или нужен chain через другое поле?

**Решение:** добавить chain `f.opts?.x ?? f.custom?.x ?? DEF_OPT.x` или фильтр `if (!f.custom)` где смысл требует.

### Баг класса 2: «регистрация после рендера»

Новый объект попадает в реактивный store, рендер начинается раньше чем shared-структура (MD/FX_FACTORS) заполнена → `MD[id] === undefined` → crash в `m.lmPer`.

**Где искать:**
- Все вызовы `emit('update', ...)` или прямой `rooms.push(...)` — есть ли регистрация до них?
- `restorePersistedState`, `loadFromHash` — регистрация ДО `rooms.splice(...)`?

**Решение:** `registerCustom(spec)` синхронно перед изменением реактивных refs.

### Баг класса 3: «потеря UI-state при reopen»

Сохранили `lmPer = 14400` (lossy число). При reopen форма должна показать «14.4 Вт/м × 10 м», но восстанавливает дефолт «9.6 Вт/м × 5 м = 4800 лм».

**Где искать:**
- `lmPer`, `area`, любые агрегаты — есть ли потеря компонент?
- Boolean-режимы (toggle типа «знаю люмены») — сохраняются?

**Решение:** `inputs?: {...}` поле на самой спеке, хранит сырой UI-state.

### Баг класса 4: «isDirty ложно-положительный»

Добавили новое опциональное поле в spec. При reopen старого объекта (где этого поля нет) `JSON.stringify(c) !== JSON.stringify(buildSpec())` → isDirty=true сразу → пользователь не может «Назад» без confirm.

**Где искать:**
- Сравнение через `JSON.stringify` — учитывает все поля включая `undefined` vs отсутствие.
- Нормализуйте обе стороны или используйте «core fields» сравнение.

**Решение:** `stripInputs(spec)` или подобный — сравниваем только содержательные поля.

### Баг класса 5: «перекошенный агрегат»

`denom` считает по фильтрованной подвыборке, `sum` — по всей. Среднее искажается.

**Где искать:**
- `Math.round(sum / denom)` — обе стороны над одной выборкой?
- `Math.round(part / total) * 100` — `part` и `total` синхронны?

**Решение:** один источник истины для выборки (`const ownFx = ...; sum/denom оба от ownFx`).

### Баг класса 6: «группировка не сливает дубли»

Каждый объект имеет уникальный id (hash или UUID). `groupBy(f.m)` создаёт по группе на объект. Текст «торшер, торшер, торшер» вместо «три торшера».

**Где искать:**
- Любая группировка по `f.m` — если id уникален per-instance, объекты НЕ сливаются.

**Решение:** для агрегации использовать содержательный ключ (тип+размер+зона), а не id.

### Баг класса 7: «UPPERCASE-плашка пустого бренда»

Поле `brand` опционально пустое. Плашка отображается с `text-transform: uppercase` — пустая полоска занимает место.

**Где искать:**
- Любая UPPERCASE / стилизованная плашка с динамическим текстом — есть ли `v-if="text"`?

**Решение:** `v-if="fxLine(it.m)"` или подобный.

### Баг класса 8: «module-level `let` сбрасывается при Vite HMR»

Хранил флаг типа «пользователь закрыл баннер до перезагрузки» в `let inSession = false` на верхнем уровне модуля. В prod-сборке это работает (let живёт пока страница в памяти). В Vite dev — HMR перезагружает модуль при любой правке другого vue-файла, флаг сбрасывается, баннер возвращается. Пользователь сообщает «не работает», хотя в prod код корректный.

**Где искать:**
- Все module-level `let`/`const` с mutable state (счётчики, toggle-флаги, кеши).
- Особенно подозрительные: «помнить до reload страницы», «единожды за сессию».

**Решение:** `document.body.dataset.someFlag` — переживает HMR (body не пересоздаётся при перезагрузке модуля), очищается только реальным reload. Альтернатива (если нужно ПЕРЕЖИТЬ reload) — `sessionStorage`.

**Прецедент:** `PWAInstallBanner.dismissedInSession` → `document.body.dataset.wlBannerDismissed`.

### Баг класса 9: «промах по дочерней кнопке внутри `<a href>`»

Кнопка-крестик внутри ссылки `<a>` — палец, чуть промазавший по 28×28 иконке, попадает на ссылку и навигирует вместо закрытия. `e.preventDefault()` в кнопке не помогает — клик физически за пределами кнопки.

**Где искать:**
- Любые маленькие интерактивные элементы внутри больших clickable родителей (`<a>`, `<button>`, `@click`-обёрток).
- Особенно crit для тачскринов где hit-area меньше 44×44 (iOS HIG минимум).

**Решение:** вынести внутренний элемент из родителя — `position: absolute` поверх, hit-area ≥44×44, визуально внутри hit-area меньший круг/иконка. См. `PWAInstallBanner.vue`.

## Шаг 5 — Финальный отчёт

Структура отчёта о проверке:

1. **Найденные баги (исправлены)** — список с приоритетом (критический/видимый/минорный), кратким описанием, файлом, фиксом.
2. **Что проверил и работает корректно** — список симулированных сценариев A-I и подтверждений.
3. **Edge cases, оставленные как есть** — с обоснованием почему не блокер.
4. **Готовность к запуску** — парсинг ok, симуляция пройдена, ждёт `npm run docs:dev` от пользователя.

## Антипаттерны (так НЕ делать)

- **Считать что «парсинг чист = всё работает».** Парсинг ловит синтаксис, не семантику. Лук-апы по undefined полям, перекошенные агрегаты, гонки регистрации — всё за пределами AST.
- **Симулировать только happy path.** Сценарии reopen-after-save, share-and-receive, restore-after-restart — где самые тонкие баги.
- **Откладывать «edge cases» без явного обоснования.** Если оставляете баг — запишите в отчёт почему: priority, влияние, временный план.
- **Полагаться на TS-типизацию для рантайм-инвариантов.** `MD[ModelId]` даёт `Model` (не `Model | undefined`), но рантайм может вернуть `undefined` если id не зарегистрирован. Все защитные `if (!m)` остаются обязательными.
