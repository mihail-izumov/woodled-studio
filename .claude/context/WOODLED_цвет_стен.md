# WOODLED — Цвет стен: пресеты, единый цвет комнаты, автоклассификация

Эта спека описывает подсистему отделки стен в конфигураторе: какие
данные хранятся, как HEX переводится в категорию `light/medium/dark`,
как это влияет на расчёт нормы и тексты карточки «Стены» в блоке
настроения.

> История:
> — Первая итерация (июнь 2026) ввела отдельное поле `wallColor` и свой
>   HEX-input в RoomSettings. UX оказался путающим: на главной уже был
>   пикер «Цвет комнаты» (`cardColor`), и пользователь не понимал, что
>   это два разных цвета.
> — Финальная итерация: **единый цвет комнаты** (`cardColor`) служит и
>   акцентом UI, и источником для физики стен. Доступ к пикеру — с двух
>   мест: с главной (RoomCard) и из параметров комнаты (RoomSettings).
>   Поле `wallColor` удалено.

---

## 1. Зачем

Физика освещения честно работает с **тремя классами отражения** матовой
поверхности (IES lumen method): светлая ≈ 70%, средняя ≈ 50%, тёмная ≈ 30%.
Цветовое пространство пользователя при этом — миллионы оттенков
(референсы из Pinterest, краска по артикулу, Figma-палитра). Чтобы дать
свободу выбора и не разрушить расчёт, цвет переводится в одну из трёх
категорий по **relative luminance (BT.709)**, а уже категория идёт в
формулу нормы.

Два уровня свободы:

| Уровень | Контроль | Где задаётся |
|---|---|---|
| Пресет (3 кнопки) | категория явная | `RoomSettings.vue`, кнопки «Светлая / Средняя / Тёмная» |
| Свой цвет (палитра / колесо / HEX) | категория автокласс | `ColorPickerModal.vue` — открывается с RoomCard ИЛИ из RoomSettings |

Если задан `cardColor` — он **перекрывает** пресет в расчётах и в
карточке настроения.

## 2. Карта файлов

| Файл | Что внутри |
|---|---|
| `data/rooms.ts` | Тип `WallFinish = 'light' \| 'medium' \| 'dark'`. Поля `Room.wallFinish?: WallFinish` (пресет — fallback) и `Room.cardColor?: string` (единый HEX комнаты, источник для физики стен). |
| **`engine/wall-color.ts`** | Физика и нормализация. `normalizeHex`, `relativeLuminance`, `wallFinishFromHex`, `wallFinishOf(room)`, `wallColorOf(room)`, константы `WALL_FINISH_THRESHOLDS` и `WALL_PRESET_HEX`. |
| `engine/brightness.ts` | `baseLm` читает `wallFinishOf(r)` и умножает UF на `WALL_UF_MULT[finish]` (light=1.1, medium=1.0, dark=0.85). |
| `engine/forest-cards.ts` | `wallsCard` собирает chip/swatch/текст из `wallFinishOf(room)` + `room.cardColor`. Банки `WALL_LIGHT / WALL_MEDIUM / WALL_DARK`. |
| `components/ColorPickerModal.vue` | Единый пикер: вкладка «Готовые цвета» (палитра), вкладка «Свой цвет» (колесо HSL + HEX-input + кнопка «Вставить»). Puck позиционируется через `wheelPosForHex` (HEX→HSL→координаты). |
| `components/RoomSettings.vue` | Три кнопки-пресета wallFinish + кнопка-плашка «Или свой цвет», открывающая локально `ColorPickerModal`. Применяет результат к `draftCardColor`, не трогая props.room до save. |
| `components/App.vue` | Глобальный `ColorPickerModal` для RoomCard (как раньше) — пишет в `cardColor` через `cfg.updateRoom`. |
| `components/ForestMood.vue` | Рендерит chip-плашку и цветной кружок 12×12 (если карточка прокинула `swatch`). |
| `engine/share.ts` | Pack/unpack: ключ `wf` (wallFinish), ключ `cc` (cardColor) — уже существовал. Отдельного `wc` (старого wallColor) нет. |

**Святая обязанность:** все обращения к категории стен должны идти
через `wallFinishOf(room)`, никогда — через `room.wallFinish` напрямую.
Прямое чтение игнорирует `cardColor` и ломает расчёт. Grep `r.wallFinish`
/ `room.wallFinish` должен быть пуст вне `wall-color.ts` и `RoomSettings.vue`
(там — драфт-стейт для трёх пресетов).

## 3. Два входа в пикер

```
┌───────────────────────────────┐
│ Главная (Live Home)           │
│                               │
│   ▼ Тап по «дверке» RoomCard  │
│     («Цвет комнаты»)          │
│                               │
└──────────────┬────────────────┘
               │
               │  App.vue → ColorPickerModal (singleton)
               │  pick → cfg.updateRoom({ ...room, cardColor })
               │
               ▼
       ┌───────────────────────┐
       │  cardColor: '#RRGGBB' │
       │  в Room               │
       └──────────┬────────────┘
                  │
                  │  читает wallFinishOf(room) в расчётах
                  │  и forest-cards (wallsCard)
                  ▼
        ─── физика и карточка «Стены» ───


┌───────────────────────────────┐
│ Параметры комнаты             │
│   (RoomSettings)              │
│                               │
│   Секция «Цвет стен»          │
│     • 3 пресета wallFinish    │
│     • Плашка «Или свой цвет»  │
│       → открывает ту же       │
│         ColorPickerModal      │
│         локально              │
│                               │
└──────────────┬────────────────┘
               │
               │  pick → draftCardColor (только драфт)
               │  Save → emit updated Room с cardColor
               │
               ▼
       (тот же путь, что и сверху)
```

Главное отличие двух входов:

- **С главной (RoomCard)** — изменение применяется мгновенно через `cfg.updateRoom`.
- **Из RoomSettings** — изменение хранится в `draftCardColor`, применяется только при «Сохранить» (через общий `emit('save', updatedRoom)`). `isDirty` отслеживает разницу. Это даёт пользователю возможность «передумать» и закрыть параметры без сохранения.

## 4. Физика классификации

### 4.1. Алгоритм

В `engine/wall-color.ts`:

```ts
function relativeLuminance(hex: string): number {
  const lin = (c: number) =>
    c <= 0.03928 ? c / 12.92 : ((c + 0.055) / 1.055) ** 2.4
  return 0.2126 * lin(r) + 0.7152 * lin(g) + 0.0722 * lin(b)
}

const WALL_FINISH_THRESHOLDS = { light: 0.55, dark: 0.20 }

function wallFinishFromHex(hex: string): WallFinish {
  const L = relativeLuminance(hex)
  if (L >= 0.55) return 'light'
  if (L <= 0.20) return 'dark'
  return 'medium'
}

function wallFinishOf(room) {
  if (room.cardColor) {
    const norm = normalizeHex(room.cardColor)
    if (norm) return wallFinishFromHex(norm)
  }
  return room.wallFinish ?? 'medium'
}
```

### 4.2. Почему BT.709 и почему именно эти пороги

- **BT.709 luminance** — стандартное приближение того, что человеческий
  глаз ощущает как «яркость». Для матовой стены luminance ≈ reflectance,
  поэтому это физически обоснованная метрика для UF, а не «среднее RGB».
- **Порог 0.55 (light)** ≈ 70% reflectance в IES — бумага, светлая краска.
- **Порог 0.20 (dark)** ≈ 30% reflectance — тёмные обои, морёный дуб.
- Всё между — `medium` (стандарт «обычная отделка»).

### 4.3. Влияние на расчёт нормы

`engine/brightness.ts`:

```ts
const WALL_UF_MULT = { light: 1.1, medium: 1.0, dark: 0.85 }
```

| Категория | UF-множитель | Что значит |
|---|---|---|
| light | ×1.1 | стены вернут +10% света → норма комнаты на 10% меньше |
| medium | ×1.0 | без поправок |
| dark | ×0.85 | стены съедят ~15% → норма комнаты на 15% больше |

```
baseLm = lux × area × (1 + Σfurniture.ab) / (UF × WALL_UF_MULT[finish] × MAINT_FACTOR)
```

### 4.4. Проверка значений

| HEX | Luminance | Категория |
|---|---|---|
| `#FFFFFF` | 1.000 | light |
| `#E8E0D4` (наш пресет «Светлая») | 0.752 | light |
| `#CCCCCC` | 0.604 | light |
| `#999999` | 0.319 | medium |
| `#8A8276` (пресет «Средняя») | 0.227 | medium |
| `#5A5045` | 0.083 | dark |
| `#2E2921` (пресет «Тёмная») | 0.023 | dark |
| `#000000` | 0.000 | dark |
| `#B85C6C` (пресет палитры «Рубин») | 0.18 | dark |
| `#5B8BA0` (пресет палитры «Озеро») | 0.23 | medium |

Реалистичные акцентные цвета из палитры `ColorPickerModal` дают
ожидаемую классификацию: красные/глубокие тёмные → dark, голубые/
средние → medium, песочные/льняные → medium-light.

## 5. Нормализация ввода (`normalizeHex`)

Принимает любую разумную форму HEX и приводит к каноническому `#RRGGBB`
в нижнем регистре. Возвращает `null`, если строка не похожа на HEX.

| Вход | Выход |
|---|---|
| `#E8E0D4` | `#e8e0d4` |
| `e8e0d4` | `#e8e0d4` |
| `#abc` | `#aabbcc` (короткий разворачивается) |
| `abc` | `#aabbcc` |
| `#E8E0D4FF` | `#e8e0d4` (alpha-канал отбрасывается) |
| `  e8e0d4  ` | `#e8e0d4` (пробелы стрипаются) |
| `qwerty` | `null` |
| `''` или `null` | `null` |

В store пишется только нормализованная форма.

## 6. UI в `ColorPickerModal.vue`

Полноэкранная модалка с двумя вкладками:

### 6.1. Вкладка «Готовые цвета»
12 акцентных пресетов в сетке 6×2 (Закат, Янтарь, Песок, Лён, Поляна,
Лес, Озеро, Небо, Лаванда, Рубин, Глина, Орех). Тап по пресету
устанавливает `picked`, синхронизирует `hexInput` и двигает puck
(через `wheelPosForHex`) — на случай перехода на вкладку «Свой цвет».

### 6.2. Вкладка «Свой цвет»
- Колесо HSL 280×280 (canvas). Сектор спирали:
  `sat(t) = 30 + 70t, lit(t) = 85 − 35t, t ∈ [0..1]`.
- Тап по колесу → читаем пиксель, ставим picked и puck на позицию тапа.
- **HEX-input** под колесом — поле ввода с placeholder `#E8E0D4 или E8E0D4`.
  - Принимает с `#` и без, короткие HEX, alpha.
  - Кнопка «Вставить» — `navigator.clipboard.readText()`. На iOS PWA
    срабатывает после жеста.
  - Под полем — подсказка: либо ошибка («Не похоже на HEX»), либо
    распознанная категория («Распознан как нейтральный тон»).
- При вводе валидного HEX (`watch hexInputNorm`) → `picked = hex`,
  `pickerPos = wheelPosForHex(hex)`. Puck едет на правильную точку
  колеса, окрашен в выбранный цвет.

### 6.3. Puck на колесе (баг и фикс)

**Баг:** раньше при открытии вкладки `pickerPos` ставился в фиксированную
«декоративную» точку (`angle=40°, dist=0.65`) с `picked = T.neutral`. То
есть кружок-указатель был окрашен в *цвет под ним* (например, ярко-зелёный),
но это никак не соответствовало «текущему» цвету комнаты — пользователь
видел «кислотный» оттенок там, где ожидал свой выбор.

**Фикс:** позиция puck вычисляется из `picked` через `wheelPosForHex`:

```ts
function hexToHsl(hex)        // → [h, s, l] (0-360, 0-100, 0-100)
function wheelPosForHex(hex)  // → { x: 0-100, y: 0-100 }
//  t = clamp(0..1, (tSat + tLit) / 2),
//  где tSat = (s − 30) / 70, tLit = (85 − l) / 35
//  x = 50 + 50·t·cos(h)
//  y = 50 + 50·t·sin(h)
```

Для цветов, **физически отсутствующих на нашем колесе** (очень тёмные
или очень светлые — например `#B85C6C` с L=0.22, S вне диапазона sat
spirali), `t` clamp-ится в [0..1]. Puck стоит на краю колеса в
правильном hue — это лучше, чем фиксированная случайная точка. Сам
кружок при этом окрашен в picked (точный HEX), и визуальный
«непоказанный» оттенок честно отображается на самом puck-е.

### 6.4. Кнопки
- **Готово** — эмитит `pick(normalizedHex)` и `close`.
- **Сбросить цвет** — `picked = undefined, pickerPos = null, hexInput = ''`.

## 7. UI в `ForestMood.vue` (карточка «Стены»)

`wallsCard` возвращает структуру:

```ts
{
  title: 'Стены',
  chip: '#E8E0D4 · светлые',    // если задан cardColor
  // chip: 'светлые'             // если cardColor пуст
  swatch: '#e8e0d4',             // только при заданном cardColor
  text: '<фраза из банка>',
}
```

В `ForestMood.vue` chip-плашка отрисовывает цветной кружок 12×12 при
непустом `swatch`. Для других карточек (где `swatch === undefined`)
кружок не рисуется — backward-compat.

Банки фраз (`WALL_LIGHT / WALL_MEDIUM / WALL_DARK`) живут в
`engine/forest-cards.ts`. По 3 фразы на категорию, выбираются через
`pick(seed)` с тем же сидом, что и остальные карточки (анти-повтор).

Полный список фраз и правила тона — в `WOODLED_словарь_карточек_настроения.md`.

## 9. Как менять / расширять

### 9.1. Подкрутить пороги категорий
Только в одном месте — `WALL_FINISH_THRESHOLDS` в `engine/wall-color.ts`.
После — обновить таблицу выше + словарь карточек.

### 9.2. Изменить эффект на расчёт
`WALL_UF_MULT` в `engine/brightness.ts`. Калиброваны под IES, менять
синхронно с порогами и текстами в `forest-cards.ts` («−10% / +15%»).

### 9.3. Поменять дефолтный HEX пресетов wallFinish
`WALL_PRESET_HEX` в `engine/wall-color.ts`. Эти HEX показываются в
превью когда `cardColor === undefined`.

### 9.4. Поменять тексты фраз
Банки в `engine/forest-cards.ts`. Перед правкой — чек-лист ToV из
`WOODLED_словарь_карточек_настроения.md`.

### 9.5. Расширить палитру в ColorPickerModal
Массив `PRESETS` в `ColorPickerModal.vue`. После добавления цвета
проверить через `wallFinishFromHex`, что он попадает в ожидаемую
категорию.

### 9.6. Свободный HEX для других UI
Если нужно, чтобы свободный цвет влиял ещё где-то — читать только через
`wallColorOf(room)` / `wallFinishOf(room)`, не дублировать парсинг HEX.

## 10. Подводные камни

- **Прямое чтение `room.wallFinish`** в коде расчётов — баг. Всегда через `wallFinishOf(room)`.
- **`isDirty` без нормализации** в RoomSettings: значения сравниваются как строки (`draftCardColor` хранит уже нормализованный HEX или undefined — потому что ColorPickerModal эмитит нормализованную форму).
- **`navigator.clipboard.readText`** требует пользовательский жест. Без активации вернёт ошибку — глотаем молча.
- **Alpha-канал.** HEX с 8 символами — alpha отбрасывается (для стен смысла нет).
- **Короткий `#abc`** разворачивается в `#aabbcc` (CSS-стандарт).
- **Регистр.** Нормализуем в нижний. В chip-виде делаем `.toUpperCase()` локально.
- **Пресет `medium` (#8A8276)** близок к границе dark (L=0.23). Сам пресет даёт enum 'medium' напрямую, поэтому пограничность не страшна. Если пользователь введёт сам этот HEX через ColorPickerModal — тоже `medium`. Чуть темнее (`#7A7066`) — уже `dark`.
- **Цвета вне HSL-колеса.** Колесо показывает только L=50–85% и S=30–100%. Для цветов вне этого диапазона (очень тёмные / десатурированные) puck стоит на ближайшей точке колеса (clamp), но цвет puck — точный picked HEX. Это компромисс: альтернатива — менять колесо целиком.

## 11. Тест-кейсы (ручной чек-лист)

1. С главной → тап «Цвет комнаты» на RoomCard → выбрать пресет «Рубин» (#B85C6C) → Готово → норма комнаты выросла ≈ на 15% (классификация = dark), в карточке «Стены» появился красный кружок и chip `#B85C6C · тёмные`.
2. С главной → Свой цвет → ввести `e8e0d4` (без `#`) → puck едет на светлую часть колеса, превью светлый цвет, категория «светлый тон». Готово → норма комнаты упала на 10%.
3. Параметры комнаты → «Или свой цвет» → открывается та же модалка → выбрать колесом → Готово → значение в draftCardColor, плашка показывает новый HEX и категорию. «Сохранить» — применяется. «Назад» без сохранения — `LeaveConfirmModal`.
4. Ввести `qwerty` в HEX-input → красная ошибка, picked не меняется.
5. Скопировать `#3A3530` в буфер → «Вставить» → подтянулся, категория «тёмный тон».
6. «Сбросить цвет» → `picked = undefined`, `hexInput = ''`, puck исчезает. После Готово — в карточке «Стены» показывается пресет wallFinish.
7. Создать share-ссылку с заданным `cardColor` → открыть в incognito → цвет восстановился, swatch в карточке.
8. Открыть пикер с уже заданным cardColor (`#B85C6C`) → перейти на «Свой цвет» → puck стоит на правильной точке колеса (красный сектор, ближе к краю), окрашен в `#B85C6C`. HEX-input заполнен `#B85C6C`.
