# WOODLED — Шаринг конфигурации по ссылке

Спека сериализации состояния в URL-хеш. Источник правды — `engine/share.ts`.
Этот документ — про что обязательно должно попадать в ссылку и как ловить
регрессии при изменении моделей данных.

> 📍 Когда читать: перед правкой `engine/share.ts`, перед добавлением нового
> поля в `Room`/`Fixture` (`data/rooms.ts`, `data/catalog.ts`, `data/materials.ts`),
> перед тем как менять алгоритм расчётов так, что он начинает читать новое поле.

## Зачем это нужно

Пользователь собирает дом и шарит его другу. Ссылка должна открыться
**идентично исходной** — те же цвета, площади, мебель, светильники, проценты
яркости. Нет бэкенда: всё состояние едет в URL-хеше.

Расчёты (`baseLm`, `fxLm`, `ratioOf`, `wallFinishOf`, `forestScene`, карточки)
**детерминированы** относительно `(Room, Fixture)` + статических каталогов
(`MD`, `FX_FACTORS`, `RTS`, `FURN`, `UF_CURVE`, `WALL_UF_MULT`). Каталоги вшиты
в код и одинаковы у отправителя и получателя в рамках одной сборки. Значит,
если в ссылке честно передаются все поля Room/Fixture, картина у получателя
сойдётся 1:1.

Если хоть одно поле, которое читает движок, **не попадает в pack** — у
получателя оно подставится дефолтом, и:
- меняется цвет / тинт UI,
- меняется UF в `baseLm` → расходятся **проценты яркости** и статус яркости,
- может сломаться имя лесной сцены (через `forestScene`),
- может уехать совет в карточке настроения (`forest-cards.ts`).

## Формат ссылки

```
#s=2.<lz-string-encoded JSON>     // дом (state)
#fx=2.<lz-string-encoded JSON>    // один светильник
```

- Префикс `2.` отделяет v2 (lz-string) от v1 (base64url-JSON).
- v1 — legacy: декодируется при чтении, не пишется.
- Точка не входит в base64url-алфавит → дискриминатор форматов однозначный.

Минимизация имён полей (`t`/`n`/`si`/`cc`/…) даёт ~2× выигрыша, lz-string —
ещё ~3–4×. На реальном state средней комнаты — ~2300 → ~600 байт.

## Карта полей: Room ↔ PackedRoom

| Источник (`Room`) | Pack-ключ | Дефолт при unpack | Что ломается если пропустить |
|---|---|---|---|
| `typeId` | `t` | — (обязательное) | Без типа нет комнаты |
| `customName` | `n` | `''` | Имя комнаты сбрасывается на тип |
| `sizeIndex` | `si` | `1` (Средняя) | Площадь и норма уходят на дефолт |
| `customArea` | `ca` | `null` | Кастомная площадь теряется → fallback на Большую |
| `ceilingH` | `h` | `2.7` | Меняется индекс помещения → UF → baseLm → ratio |
| `wallFinish` | `wf` | `'medium'` | Fallback физики стен; теряется только явный пресет |
| `cardColor` | `cc` | `undefined` | **Цвет комнаты + цвет стен**. Без него: тинт UI сваливается на `T.neutral`; `wallFinishOf` пересчитывает категорию по дефолту → меняются UF, baseLm, ratio, %. |
| `fixtures` | `fx` | `[]` | Светильники теряются |
| `furniture` | `fu` | `[]` | Мебель и её поглощение света теряются |
| `limits` | `lim` | `{}` | Лимиты зон сбрасываются → `addFixture` пускает любое количество |

## Карта полей: Fixture ↔ PackedFixture

| Источник (`Fixture`) | Pack-ключ | Дефолт при unpack |
|---|---|---|
| `m` (модель) | `m` | — (обязательное) |
| `q` (количество) | `q` | `1` |
| `wood` | `w` | `'oak'` |
| `zone` | `z` | `'ceiling'` |
| `l` (override ламп) | `l` | `undefined` (модельный `lamps`) |
| `opts` (`FxOpts`) | `o` | `undefined` |
| `done` (чек-лист) | `d` | `undefined` |

`o` типизировано как `Partial<FxOpts>` — это автоматически захватывает
все поля опций (`bowl`, `mount`, `wire`, `btemp`, `diffuser`, `moisture`,
`bulbs`, `bulbOpt`, `baseColor`). Если в `materials.ts` появится новая
опция — она поедет в ссылку без правок `share.ts`.

## Жёсткое правило при изменении моделей

При **добавлении или переименовании поля** в `Room` (`data/rooms.ts`) или
`Fixture` (`data/catalog.ts`), а также при добавлении поля в `FxOpts`
(`data/materials.ts`):

1. **Добавить в `PackedRoom`/`PackedFixture`** короткий ключ (2–3 буквы).
2. **Обновить `packRoom`/`packFixture`** — записывать только если значение
   отличается от дефолта (экономим байты, длинная ссылка → UI криво).
3. **Обновить `unpackRoom`/`unpackFixture`** — подставить тот же дефолт,
   что и при создании комнаты/светильника.
4. **Прогнать roundtrip-тест** (см. ниже): `packRoom(r); unpackRoom(packRoom(r)) ≡ r` для всех значимых полей.
5. **Учитывать совместимость со старыми ссылками**: если поле новое — у старой
   ссылки его не будет, должен сработать тот же дефолт. Если поле меняет
   смысл (например, `wallColor` слили в `cardColor`) — нужна миграция при
   `unpackRoom` или явный bump формата (v3).

## Что НЕ должно попадать в ссылку

- UI-флаги: `active`, `showStory`, `showBuy`, `showShare`, `picker`, `firstId`,
  `welcomeSeen`, `dashboardTourSeen`, `onboardedOnce`, тосты — локальное
  состояние клиента.
- `id` комнаты — регенерируется через `nextId` при unpack (иначе коллизия с
  существующими комнатами получателя).

## Совместимость и миграции

- v1 (legacy base64url-JSON) декодируется в `decodePayload` без префикса.
- v2 — текущий формат.
- При радикальной смене семантики поля — bump до v3 (`'3.'` prefix),
  `decodePayload` маршрутизирует по префиксу.
- Изменение **статических каталогов** (`FX_FACTORS`, `UF_CURVE`, `MD`,
  пороги `BRIGHT`) — это другая ось. Это не серилизация, а версия алгоритма.
  Если разнести версии бандла у отправителя и получателя — получится разный
  расчёт от одинаковой ссылки. Решается принудительным апдейтом бандла
  (reload-кнопка, cache-bust), а не сериализацией.

## Как проверять (roundtrip)

Дешёвый sanity-чек без сборки проекта — копия `packRoom`/`unpackRoom` в
node-скрипт + сравнение `JSON.stringify(original)` с `JSON.stringify(restored)`
по списку полей. Шаблон — см. `outputs/test-share.mjs` (использовался при
фиксе `cardColor`).

При правке `share.ts` обязательно гонять roundtrip как минимум на:
- комнате со всеми полями (включая `cardColor`, `customArea`, `wallFinish='dark'`, `ceilingH≠2.7`);
- дефолтной комнате (проверить, что лишние ключи не лезут в payload);
- светильнике с `q>1`, `opts`, `done`.
