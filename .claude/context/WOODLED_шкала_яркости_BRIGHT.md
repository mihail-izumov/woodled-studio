# WOODLED — шкала яркости BRIGHT (источник правды)

> Документ фиксирует систему статусов яркости: что это, зачем, как считается, почему именно такие пороги, где живёт в коде и что нельзя ломать. Читать перед правкой `moods.ts`, `copy.ts`, `brightness.ts`, `BrightnessArc.vue`, `RoomCard.vue`, `FxEditor.vue`, `SmartHelpModal.vue`.

---

## 0. TL;DR

Шкала BRIGHT переводит ratio (факт/норма люменов) в одно слово на бейдже. Пять статусов — от «Темно» до «Праздник». Пороги обоснованы профстандартами (IES, СП 52.13330). Шкала универсальная для всех комнат — per-room адаптация уже заложена в саму норму (baseLm). Характер света (откуда, какой) — отдельная система (лесные сцены), не дублировать в BRIGHT.

---

## 1. Финальная шкала

| ratio | Статус | Цвет | Тон (ToV) |
|---|---|---|---|
| ≤ 0.5 | **Темно** | `T.red` | забота, без юмора |
| ≤ 0.8 | **Полусвет** | `T.yellow` | мягко, по-доброму |
| ≤ 1.2 | **Светло** | `T.green` | спокойное одобрение |
| ≤ 1.5 | **Ярко** | `T.neutral` | констатация |
| > 1.5 | **Праздник** | `T.textSec` | подмиг (юмор допустим) |

---

## 2. Обоснование порогов

### 2.1. Почему эти числа (профессиональные стандарты)

Модель WOODLED считает норму (`baseLm`) методом коэффициента использования (СП 52.13330 / IES lumen method) с учётом: типа комнаты (lux), площади, высоты потолка (индекс помещения), отделки стен, мебели, КПД корпуса (`FX_FACTORS`), свеса подвеса и коэффициента эксплуатации MF=0.8. Это значит: **ratio=1.0 = ровно столько света, сколько нужно, уже с запасом на старение ламп и пыль**.

- **IES (международный стандарт):** допустимое отклонение от цели — **+10%**. Значит ratio 1.1 — верхняя граница «нормы» по строгому стандарту.
- **СП 52.13330.2016 (Россия):** коэффициент запаса при проектировании — **1.3–1.5**, но он предназначен для компенсации деградации. В нашей модели MF=0.8 уже это учитывает. Ratio=1.5 в нашей системе — это 1.5× от нормы, которая САМА включает запас.
- **Практика:** overlighting — признанная проблема. Избыточная яркость вызывает дискомфорт, блики и нагрузку на глаза (IES, WELL Building Standard).

### 2.2. Почему именно 0.5 / 0.8 / 1.2 / 1.5

| Порог | Обоснование |
|---|---|
| **0.5** (Темно → Полусвет) | Менее половины честной нормы = объективно темно для любой комнаты. |
| **0.8** (Полусвет → Светло) | Совпадает с порогом настроений (Утро→Полдень). 80% нормы = заметный дефицит, но MF=0.8 означает «хватит для новых ламп, нет запаса на старение». Граница функционально достаточного света. |
| **1.2** (Светло → Ярко) | IES допускает +10%, мы даём +20% — щедрый коридор для вкуса. Полная многослойная сборка (люстра + бра + торшер) садится в верхнюю часть этой зоны (~1.19). |
| **1.5** (Ярко → Праздник) | Край проектного запаса по СП (1.3–1.5). Всё выше — overlighting по профстандартам. В нашей модели MF уже учтён, значит 1.5× = реально в полтора раза больше, чем нужно. |

### 2.3. Проверка на эталонной сборке (спальня 20 м², норма ~5949 лм)

| Сборка | ratio | Статус |
|---|---|---|
| 1× Rotor M | 0.66 | **Полусвет** — одной люстры мало, мотивирует добавить |
| + 2 бра | 1.02 | **Светло** — базовый набор собран |
| + торшер | 1.19 | **Светло** — добавление не наказывается |
| + 2 настольные | 1.40 | **Ярко** — полный вечерний набор, честный сигнал |
| Перебор | 1.72 | **Праздник** — реально много, можно приглушить |

### 2.4. История изменений порогов

- **v1 (старая модель, норма занижена ~2×):** 0.5 / 0.8 / 2.0 / 4.0. Названия: Мало света / Почти хватает / В самый раз / С запасом / Больше, чем надо. Широкие полосы компенсировали неточную норму.
- **v2 (первая итерация после честной нормы):** 0.5 / 0.8 / 1.5 / 2.5. Те же названия. Полосы сужены, но верх оставался щедрым.
- **v3 (финальная, текущая):** 0.5 / 0.8 / 1.2 / 1.5. Новые названия (Темно/Полусвет/Светло/Ярко/Праздник). Обоснованы IES и СП.

## 3. Принципы системы (не ломать)

### 3.1. Разделение ролей — три слоя, три вопроса

| Слой | Вопрос | Ответ | Файл |
|---|---|---|---|
| **BRIGHT** (бейдж) | Сколько света? | Темно / Полусвет / Светло / Ярко / Праздник | `moods.ts` |
| **Сцена** (настроение) | Какой свет, откуда? | Ореховая роща / Дубовая чаща / Вековая поляна… | `forest.ts` |
| **Copy** (подсказка) | Что делать? | «бра у изголовья просится» / «можно приглушить» | `copy.ts` |

**BRIGHT не оценивает, как ощущается свет в пространстве** — это задача сцены. BRIGHT — чистая шкала яркости (темно↔ярко), как термометр. Сцена — про характер (откуда свет, какое дерево, какая температура). Подсказка — про действие.

### 3.2. Универсальная шкала, per-room норма

Пороги BRIGHT одинаковые для всех комнат. Per-room адаптация уже встроена в `baseLm`:
- Кухня: lux=200, спальня: lux=100, кабинет: lux=250
- Плюс: площадь, потолок (индекс помещения RI), отделка стен (WALL_UF_MULT), мебель (Σab), подвес (SUSPENSION_DROP), эксплуатация (MF=0.8)

Поэтому ratio=0.8 в спальне и ratio=0.8 на кухне — разные абсолютные люмены, но одинаковая доля от потребности. Универсальный порог работает корректно.

### 3.3. Названия — шкала яркости, не оценки

Старые названия («Почти хватает», «В самый раз», «С запасом») содержали оценку (хватает/не хватает). Новые — чистые дескрипторы яркости: Темно → Полусвет → Светло → Ярко → Праздник. Оценочность оставлена сцене и подсказкам.

Исключение — **«Праздник»**: намеренный слом паттерна (как Tesla Ludicrous). Слово-подмиг для экстремального режима, по ToV допустимо (юмор в «хорошем» состоянии).

### 3.4. Тон под состояние (из ToV)

| Статус | Тон | Юмор |
|---|---|---|
| Темно | Забота, конкретный шаг | Нет |
| Полусвет | Мягко, по-доброму | Чуть-чуть |
| Светло | Спокойное одобрение | Да, можно подмигнуть |
| Ярко | Лёгкая констатация | Да |
| Праздник | Подмиг, мягкая гипербола | Да |

## 4. Где это живёт в коде

### 4.1. Карта файлов

| Файл | Что делает | Что менять |
|---|---|---|
| `data/moods.ts` → `BRIGHT[]` | Массив порогов + названия + цвета | Пороги `max`, названия `name` |
| `data/moods.ts` → `getBright(ratio)` | Возвращает статус по ratio | Не трогать (читает BRIGHT) |
| `engine/copy.ts` → `lightState(ratio)` | Возвращает `dark/dim/ok/plenty/over` | Пороги синхронно с BRIGHT |
| `engine/copy.ts` → `lightHint(...)` | Текст подсказки per-room | Банки текстов |
| `engine/brightness.ts` → `baseLm()` | Норма (UF × MF метод) | Калибровка UF_CURVE, MAINT_FACTOR |
| `engine/brightness.ts` → `fxLm()` | Сумма факт. люмен (с FX_FACTORS) | — |
| `components/ui/BrightnessArc.vue` | Полукруглый gauge на Canvas | Визуальные уровни по brightIdx |
| `components/RoomCard.vue` | Бейдж на карточке + прогресс-солнце | `pct = min(100, round(ratio×100))` |
| `components/RoomDetail.vue` | Оркестратор: считает ratio, раздаёт | — |
| `components/FxEditor.vue` | Рекомендации размера, сравнивает имена | Строковые сравнения с названиями |
| `components/ui/SmartHelpModal.vue` | Модалка «WOODLED Smart» | Тексты, описания статусов |

### 4.2. Поток данных

```
data/rooms.ts (lux, sizes)  ──┐
data/catalog.ts (FX_FACTORS) ──┤
data/furniture.ts (ab)       ──┤
                               ▼
engine/brightness.ts:
  baseLm(rt, room) → норма (план)
  fxLm(fixtures)   → факт (сумма)
                               │
                               ▼
         ratio = факт / план (или 0)
                               │
         ┌─────────────────────┼──────────────────┐
         ▼                     ▼                   ▼
  moods.ts:              copy.ts:            forest.ts:
  getBright(ratio)       lightState(ratio)    forestScene(rt, room)
  → { name, color }     → dark/dim/ok/…      → { name, legend }
         │                     │                   │
         ▼                     ▼                   ▼
  BrightnessArc         lightHint(...)        ForestMood.vue
  RoomCard бейдж        подсказка             имя сцены + карточки
  FxEditor бейджи
```

### 4.3. Синхронизация порогов (критично!)

Пороги в `BRIGHT` (moods.ts) и `lightState` (copy.ts) ДОЛЖНЫ совпадать:

```
BRIGHT:     0.5  →  0.8  →  1.2  →  1.5  →  ∞
lightState: 0.5  →  0.8  →  1.2  →  1.5  →  ∞
             dark    dim     ok     plenty   over
```

Если рассинхронить — бейдж покажет одно, а подсказка скажет другое.

### 4.4. FxEditor: строковые сравнения (осторожно!)

`FxEditor.vue` (рекомендации размера) сравнивает `getBright(projectedRatio).name` со строками: `'Темно'`, `'Полусвет'`, `'Праздник'`, `'Ярко'`. При переименовании статусов — обновлять и эти строки. Grep: `selBright===`.

## 5. Прогресс-бары: как отображается ratio

### 5.1. BrightnessArc (страница комнаты)

Полукруглый Canvas-gauge. Угол прогресса = `min(ratio, 1) × 180°`. Кап на ratio=1.0 (100%). Выше 100% дуга полная, различаются только статус-текст и визуальный уровень «солнца» (5 ступеней по brightIdx).

### 5.2. RoomCard (главная, солнце на линии)

`pct = min(100, max(0, round(ratio × 100)))`. Кап на 100%.

**Баг (исправлен):** ранее было `ratio / 2 × 100` — при 50% света солнце стояло на 25% линии. Убрано деление на 2.

### 5.3. Общая логика

Оба прогресс-бара показывают «сколько процентов от нормы набрано», с капом на 100%. Выше 100% — бар на максимуме, различие только в тексте статуса. Это корректно: бар отвечает на «достигнута ли норма», статус — на «какой это уровень яркости».

## 6. SmartHelpModal — структура

Модалка «WOODLED Smart — как мы считаем свет» объясняет систему пользователю. Структура:

1. **Сколько нужно (норма)** — тип комнаты, площадь, потолок, стены, мебель. Заменяет старое «100 лм/м²».
2. **Сколько доходит (отдача)** — КПД корпуса (ламели Rotor гасят часть), бра в стену, настольная локально. НОВЫЙ блок.
3. **Что значит шкала** — 5 статусов с описаниями.
4. **Имя сцены — про другое** — снимает путаницу «два индикатора»: шкала = сколько, сцена = откуда.
5. **Кнопка «Выбери за меня»** — алгоритм подбора размера (из FxEditor).

Сырые коэффициенты (×0.65, ratio, UF) в UI **не показываются** — только человеческие формулировки.

## 7. Связь с другими системами

### 7.1. Настроения (лесные сцены) — НЕ дублировать

Сцена (роща/чаща/поляна) отвечает на «какой свет, откуда». BRIGHT — на «сколько». Это разные вопросы. Сцена не зависит от порогов BRIGHT и наоборот. Пороги настроений (autoMood: 0.8/1.4) — отдельная система, не трогать при правке BRIGHT.

### 7.2. copy.ts (подсказки) — синхронизировать

`lightState` в copy.ts маппит те же пороги в `dark/dim/ok/plenty/over`. `lightHint` собирает per-room подсказку из банков. При изменении порогов BRIGHT — обновлять lightState синхронно.

### 7.3. Autosize (рекомендация размера) — подхватывает

`autosize.ts` использует `getBright(projectedRatio)` для прогноза бейджа при выборе размера. Подхватывает новые пороги автоматически.

5. Комментарии в `BrightnessArc.vue` lvl[] — **совпадают** с названиями?
