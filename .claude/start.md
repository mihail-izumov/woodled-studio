# 🚀 Старт работы с WOODLED Studio

Этот файл — первое, что читает AI-ассистент после того, как получил доступ к папке репозитория. Здесь — что сделать в первые 30 секунд и какие правила держать в голове всегда.

> Проект — **WOODLED Studio**, конфигуратор света (VitePress + Vue 3). Продукт в `.vitepress/customizer/`.

## Чек-лист на старт

1. Прочитай **`CLAUDE.md`** в корне репо — карта проекта, конвенции, подводные камни.
2. Прочитай **`.claude/context/README.md`** — индекс базы знаний с таблицей «когда что подтягивать».
3. **Не читай** остальные документы из `.claude/context/` заранее — подтягивай по мере необходимости, когда задача потребует. Экономь контекст.
4. Скажи коротко «готов» — и жди задание.

## Правила в голове (всегда)

- **В UI-текстах** никогда не используй `MD[m].collection` («Rotor», «Rotor X», «Rotor Elliptical») и `MD[m].name` («Rotor 1000»). Только `MD[m].type` («люстра», «бра», «торшер», «настольная», «спот») + `MD[m].chip` (размер — для люстр). Спека — `.claude/context/NAMING_SPEC.md`.
- **Пороги яркости** синхронны во всех слоях: `BRIGHT` (`moods.ts`), `lightState` (`copy.ts`), `brightLevel` (`forest.ts`, `forest-cards.ts`) — `0.5 / 0.8 / 1.2 / 1.5`. Менять только одновременно.
- **Тексты** — по `WOODLED_tone_of_voice.md`: «вы», коротко, глаголами, без AI-штампов («графичный», «окутывает», «симфония», сравнений «как…»). Перед сложными правками — чек-лист из `WOODLED_словарь_карточек_настроения.md`.
- **Падежи** в фразах карточек: «у/от/без X» → родительный (`fxNameGen` + `FURN_GEN`); «к X» → дательный (`fxNameDat`); иначе — именительный (`fxNameNom`).
- **Песочница без `node_modules`** — собирать/типчекать в ней нельзя. После правок пользователь прогоняет `npm run docs:dev` локально сам.
- **glow-распределение** в `zone-engine` / `RoomDetail.glowLayers` — не трогать при UI-правках.
- **Старый mood-модуль** (`MOODS`/`autoMood`/`moodDesc`/`MoodBlock.vue`/`MoodDetailModal.vue`) — мёртвый, заменён лесными сценами (`forest.ts` + `forest-cards.ts`). Не оживлять.

## Git-workflow (важно)

Репозиторий лежит на Mac пользователя, синхронизация с GitHub — через **GitHub Desktop** (или `git` GUI). Пользователь сам делает commit и push.

Твоя ответственность:
- Правки кода — файлами через `Edit` / `Write`. **Никаких** `git commit`/`push` от тебя.
- В конце каждой пачки правок (или законченного куска работы) выдай предложение для коммита **двумя блоками**:
  1. **Название** — одной строкой в формате [Conventional Commits](https://www.conventionalcommits.org/): `тип(скоуп): краткое описание`. Типы: `feat` / `fix` / `chore` / `docs` / `refactor` / `style`.
  2. **Описание** — 3-5 строк маркдаун-списком: что именно поменялось и зачем (не пересказ диффа, а смысл).

Пример:

```
feat(forest-cards): совет по мебели на основе FURN_HINT

- furnitureCard теперь учитывает доминирующую мебель через FURN_HINT
- если соответствующая зона пуста — предлагает добавить источник
- если занята — отмечает «уже на месте»; падежи через FURN_GEN
```

## Стиль ответа

Коротко и по делу. Без преамбул («Я сейчас сделаю…», «Давайте подытожим…», «Отлично, поехали!»). Сразу к действию или результату. Если задача неоднозначная — задай 1-2 коротких уточняющих вопроса.
